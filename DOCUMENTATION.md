# MyPomo: Project Documentation

## 1. Executive Summary
**MyPomo** is a high-performance, minimalist Pomodoro focus timer designed for users who value aesthetic clarity and data persistence. Originally conceived as a local tool, the project has been re-engineered into a production-ready, cloud-native application featuring secure multi-user authentication and a global database.

---

## 2. Requirements & Objectives
*   **Secure Authentication:** Seamless "Single Click" login via Google OAuth.
*   **Global Persistence:** Data must survive device changes and browser clears (migrated from SQLite to Supabase).
*   **Dynamic Categories:** Users must be able to define, color-code, and manage their own focus "Life Areas."
*   **Deep Reflection:** Post-session reflection flow to track goal achievement and blockers.
*   **Performance:** Sub-second page loads and instantaneous state updates via Next.js Server Actions.

## 4. Key Business & Design Decisions

### 4.1. Migration to Supabase (Cloud-First)
> [!NOTE]
> **Decision:** Abandoned local `mypomo.db` (SQLite) in favor of Supabase PostgreSQL.
> **Rationale:** Local databases are fragile and non-portable. By moving to the cloud, user focus data becomes a permanent asset accessible from any device.

### 4.2. NextAuth v5 Beta Implementation
> [!IMPORTANT]
> **Decision:** Used the unstable beta version of Auth.js instead of the stable v4.
> **Rationale:** Next.js 15+ and the App Router have fundamental architectural differences. V5 was chosen to "future-proof" the project against upcoming breaking changes in the React ecosystem.

### 4.3. Data Ownership Security (Composite Uniques)
> [!CAUTION]
> **Decision:** Implemented `@@unique([userId, id])` constraints at the database level.
> **Rationale:** In a shared database, a malicious user could theoretically guess a Category ID to delete it. By requiring both a valid Session User ID and the Record ID for every write operation, we ensure absolute data isolation.

### 4.4. Removal of Default Categories
> [!TIP]
> **Decision:** Removed "seeding" logic for default categories.
> **Rationale:** User interviews suggested that "Work" and "Personal" were too generic. The app now starts as a blank canvas, empowering users to define their own workflow from second one.

## 5. Development & Deployment Flow

### 5.1. Development Workflow
1. **Schema Definition:** Changes are first modeled in `prisma/schema.prisma`.
2. **Local Sync:** `npx prisma db push` is used to instantly update the Supabase development instance.
3. **Action Logic:** Server Actions in `src/app/actions.js` are updated to reflect new data shapes.
4. **UI Iteration:** Tailwind-free CSS ensures strict adherence to the glassmorphic design system.

### 5.2. Deployment Pipeline
MyPomo uses a streamlined CI/CD pipeline:
1. **Source:** Code is pushed to `github.com/balu7-alttab/MyPomo`.
2. **Trigger:** Vercel detects the push via its GitHub App.
3. **Build Script:** Vercel executes `prisma generate && next build`. 
    *   *Note: `prisma generate` is explicitly added to the build script to prevent "Outdated Client" errors during Vercel's caching phase.*
4. **Environment:** Vercel injects secret keys (`GOOGLE_CLIENT_SECRET`, etc.) during the build.
5. **Live:** The new version goes live at the custom Vercel domain.
